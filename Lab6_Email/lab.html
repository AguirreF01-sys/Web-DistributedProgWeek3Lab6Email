Lab 6: PyMail — Client-Server Email Simulation

Objective: Complete a working email system by implementing the send-email feature across both the client and the server. The client must collect the message and transmit it; the server must receive, validate, and deliver it.

Background

Lab 5 showed a server that executes a single function and disconnects. Real servers maintain a session — a continuous back-and-forth with a logged-in user. This lab adds two important ideas:

  • Sessions: the server loops, serving menu requests until the user quits.
  • Threading: ThreadingTCPServer handles multiple clients at the same time, so John and Beth can both be logged in simultaneously.

How the Protocol Works

The server and client use two special sentinel lines to coordinate who acts next.

SentinelSent byMeaning
__READY__ Server Send me one line of keyboard input
__COMPOSE__ Server Collect a full email (To / Subject / Body) and send all three lines

The client's main loop watches for these sentinels and reacts accordingly: 

if text == "__READY__":
    user_input = input()              # one line back to the server
    wfile.write(...)
elif text == "__COMPOSE__":
    compose_email(wfile)              # your TODO: collect and send the email
else:
    print(text)                       # normal output — just display it

Send-Email Conversation Flow

Client (john@email.com) Server ──────────────────────────────────────────────────────────── [connects] ──────> [accepts, spawns thread] <────── "Email address:" <────── __READY__ "john@email.com" ──────> <────── "Welcome, john@email.com!" <────── menu <────── __READY__ "2" (Send Email) ──────> <────── __COMPOSE__ [compose_email() runs] "beth@email.com" ──────> (recipient) "Lunch tomorrow?" ──────> (subject) "Are you free at noon?" ──────> (body) <────── "Delivered to beth@email.com!" <────── menu <────── __READY__ "3" (Quit) ──────> <────── "Goodbye!" [disconnects]

Notice that when __COMPOSE__ arrives, the client takes over — it collects all three fields locally and sends them without any further prompts from the server. The server simply waits for those three lines to arrive.

Step 3 — Test your implementation
  1. Restart the server.
  2. Connect as john@email.com and send a message to beth@email.com.

Step 4 — Two clients at once
Because the server uses ThreadingTCPServer, both clients can be connected simultaneously. While Beth is already logged in, have John send her another message. Switch back to Beth and view her inbox — the new message is there instantly. This is thread-safety in action: store_lock ensures the two threads cannot corrupt message_store at the same time.

Key Concepts

ConceptWhere you see it
Session loop handle() — loops on the menu until the user quits
Threading ThreadingTCPServer — each client gets its own thread
Shared state message_store — all threads read and write this dict
Thread safety store_lock — a threading.Lock that prevents races
Protocol sentinels __READY__ and __COMPOSE__ — coordinate client/server turns
Client responsibility compose_email() — the client owns its own UI for composing
Server responsibility send_email() — the server owns validation and storage
